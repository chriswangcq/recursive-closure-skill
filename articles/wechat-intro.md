# AI 做了 80% 就停了？我写了个状态机治它

> 让 AI Agent 真正解决复杂问题，而不是做到 80% 告诉你"已完成"。

AI Agent 最大的问题不是不够聪明，而是**做不完**。

让它重构一个模块，它能写出漂亮的代码——但测试没跑、边界没覆盖、旧文件散落在三个目录里。做了 80%，剩下 20% 的缺口永远不补。你问它"完成了吗"，它说"已完成"。

根本原因：LLM 擅长判断和执行，但不擅长**流程控制**。它没有一个机制来保证"问题真的解决了"。

所以我写了一个状态机来管它。上周拿它迁移了 651 个文档——归档旧文档、调研新结构、重写 30 个新文档、跑 4 套 CI 脚本。Agent 自动拆成 5 个子问题，每个子问题走完整流程：创建方案 → 执行 → 记录结果 → 独立验证。

最终产出：14 个问题、14 个工单、14 个结果、14 个检查。全部递归闭环，零遗漏。中间它自己发现 CI 脚本需要更新，自己派生了一个子问题去修，修完继续。从敲下命令到台账全绿，我没介入过一次。

## 模型越来越强，为什么 Agent 还是不靠谱？

最近行业里有个共识正在收敛：**模型本身不等于 Agent。**

模型的能力确实在飞速提升——coding、planning、长上下文——但你让它做一个复杂项目，它还是会跳步、会自欺、会留尾巴。原因很简单：模型解决的是"怎么做"，但没有人管"做没做完、做对没有"。

这就是为什么整个行业开始讨论模型之外的控制层——有人叫 Harness，有人叫 AgentOS——本质上是同一个问题：**怎么让一个很聪明但不够自律的模型，在复杂任务中可靠地交付结果。**

我的回答是：给它一个递归闭环的状态机。

## 核心原理：递归闭环

这套东西的核心是一个递归算法：

```
solve(问题):
  工单 = 创建方案工单(问题)
  分类工单为 one_go 或 split

  if one_go:
    结果 = 执行工单(工单)
  else:
    拆分为子问题
    子结果 = [solve(每个子问题)]
    结果 = 汇总(子结果)

  记录结果(工单, 结果)

  while True:
    检查 = check_success(问题, 结果)
    if 检查通过:
      break
    else:
      新结果 = solve(检查.跟进问题)
      结果.append(新结果)
```

看起来简单，但几个关键设计让它和普通 Agent workflow 有本质区别：

### 1. 工单完成 ≠ 问题解决

工单完成只意味着执行产出了一个结果。问题是否真正解决，由独立的 `check_success` 判定——要映射每条标准到证据、做压力测试、评估残余风险。这避免了 Agent "自己做自己批"的自欺陷阱。

### 2. 三种子问题，来源清晰

- **split（计划时拆分）**：复杂问题拆成子问题，各自递归求解
- **spawn（执行时派生）**：执行过程中发现"得先解决另一个事"，阻塞式派生子问题
- **follow-up（检查时跟进）**：验证发现缺口，创建跟进问题修补

每种子问题都有明确来源，不会凭空冒出来。split 是计划，spawn 是运行时子程序调用，follow-up 是验证后的修补。三种语义不混淆。

### 3. 状态机护栏

Agent 不决定流程——CLI 状态机决定。Agent 每走一步都要问 `ledger.py next`："下一步该做什么？"状态机根据当前台账状态返回唯一正确的下一步动作。

非法操作直接被拒绝：没定义工单就不能分类，没记录结果就不能做成功检查，子问题没关闭就不能关父问题。

工单状态机：`created → defined → classified → executing/splitting → done`

问题状态机：`todo → doing → checking → done`，可以 `→ blocked`、`→ followup`，但 `done` 是终态——只有通过 success check 才能到达。

### 4. 四角色分离

| 角色 | 职责 |
|------|------|
| 状态机（CLI） | 流程控制、ID 分配、状态转换、合法性校验 |
| Root Agent | 循环驱动：调用 `next` → 分派工作 → 记录输出 → 再调 `next` |
| Worker Agent | 专注当前动作：写工单、执行、写结果、做检查 |
| Markdown body | 持久化语义内容：问题定义、方案、证据、检查报告 |

LLM 做它擅长的（判断、写作、执行），状态机做 LLM 不擅长的（流程控制、规则执行）。今天 Root Agent 和 Worker Agent 通常是同一个 LLM，角色分离是为了未来可以把工作分派给多个 Agent 而不改状态机。

### 5. Effort 力度调节

一个旋钮控制四个维度：

| 力度 | 分类倾向 | 检查深度 | 拆分粒度 | 执行严格度 |
|------|----------|----------|----------|------------|
| low | 倾向 one_go | 轻量 | 粗粒度 | 快速 |
| medium | 平衡 | 标准 | 适中 | 标准 |
| high | 倾向 split | 深入 | 细粒度 | 严格 |
| extra-high | 全部 split | 完整审计 | 单一职责 | 最严格 |

为什么不做三个独立旋钮？因为分类、检查、拆分三者耦合——`split` 倾向天然需要更深的检查和更细的粒度。一个旋钮避免了组合爆炸。

> **让 AI 更好用的关键不是让模型更聪明，而是给它一套不允许偷懒的流程。**

## 用起来什么样

安装后，在 Claude Code 对话中一句话调用：

```
/recursive-closure-skill 重构认证模块，分离会话和令牌逻辑
```

![Demo](../assets/demo.gif)

Agent 会自动进入递归闭环：初始化台账 → 创建工单 → 分类 → 执行/拆分 → 记录结果 → 成功检查 → 完成。你只需要描述问题，剩下的全自动。

力度和语言都可调：

```
/recursive-closure-skill --effort extra-high --language zh 审计整个支付系统
```

**工作流程示例：** 以"重构认证模块"为例——

1. Agent 创建根问题，描述问题和成功标准
2. 写方案工单，包含问题定义、拟议方案、验收标准、验证计划
3. 判断为 `split`（需要拆分），创建子问题：分离会话逻辑、分离令牌逻辑、更新测试
4. 逐个解决子问题，每个子问题走完整的 工单→执行→结果→检查 流程
5. 子问题全部关闭后，记录父工单汇总结果
6. 独立验证原始问题是否真正解决——映射每条标准到证据、压力测试、残余风险
7. 通过则闭环；不通过则创建跟进问题，继续递归

全程每一步都有 Markdown 记录，可审计、可回溯。执行过程中还可以用仪表板实时查看进度——问题树、进度环、时间线，一目了然。

## 安装

最快的方式：把下面这句话粘贴给你的 Claude Code：

> Clone https://github.com/chriswangcq/recursive-closure-skill and install it to your skills directory.

手动装：

```bash
git clone https://github.com/chriswangcq/recursive-closure-skill ~/.claude/skills/recursive-closure-skill
```

除了 Claude Code，还支持 Codex CLI、Cursor、Windsurf、Cline、Roo Code——详见 GitHub README。

**什么时候该用：** 复杂、重要、多步骤的任务——重构模块、迁移数据、审计系统、编写完整文档。简单任务不需要，直接让 Agent 做就行。
